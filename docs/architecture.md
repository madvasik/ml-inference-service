# Architecture

## Компоненты

- `backend` — основной REST API на `FastAPI`
- `postgres` — хранение пользователей, моделей, предсказаний, платежей и транзакций
- `redis` — брокер и backend для Celery
- `celery` — воркер асинхронных предсказаний
- `celery-beat` — ежемесячный пересчет loyalty tiers
- `streamlit_dashboard` — админская аналитическая панель
- `prometheus` и `grafana` — наблюдаемость и дашборды

## Backend layout

## Ключевые потоки

### Предсказание

1. Пользователь загружает `pkl` модель.
2. API валидирует модель и сохраняет метаданные в БД.
3. `POST /api/v1/predictions` создает запись со snapshot стоимости.
4. Запрос уходит в `Celery`.
5. Воркер загружает модель, выполняет inference и только после успешного результата атомарно списывает кредиты.
6. Результат и статус сохраняются в БД.

### Биллинг

1. Пользователь вызывает `POST /api/v1/billing/payments`.
2. API создает подтвержденный платеж и атомарно начисляет кредиты.
3. Успешное предсказание пишет debit-транзакцию.
4. Для предсказания debit идемпотентен: повторное исполнение не создает второе списание.

### Loyalty

- Уровни: `none`, `bronze`, `silver`, `gold`
- Правила лежат в таблице `loyalty_tier_rules`
- `celery-beat` раз в месяц пересчитывает уровень по числу успешных предсказаний за прошлый месяц
- Скидка фиксируется в момент создания prediction и не меняется задним числом

## Readiness

- `/health` проверяет:
  - доступность API,
  - соединение с БД,
  - готовность схемы БД.
- Если БД доступна, но не применены миграции или отсутствуют обязательные таблицы, сервис считается `unhealthy`.
